Lifecycles of React

2018-07-09
#Work

  • When a component is rendered the first time:
    1) getDefaultProps
    2) getInitialState
    3) componentWillMount
    4) render
    5) componentDidMount

  • When a component is rendered the second time or re-rendered:
    1) getInitialState
    2) componentWillMount
    3) render
    4) componentDidMount

  • When a component is unmounted and destroyed:
    1) componentWillUnmount

  • When props change:
    1) componentWillReceiveProps
    2) shouldComponentUpdate
    3) componentWillUpdate
    4) render
    5) componentDidUpdate

  • When state changes:
    1) shouldComponentUpdate
    2) componentWillUpdate
    3) render
    4) componentDidUpdate

Notes:

  1. Mounting:
    When an instance of a component is being created and inserted into the DOM.
  2. Updating:
    An update can be caused by changes to props and state.
  3. Unmounting:
    When a component is being removed from the DOM.
  4. Error Handling:
    When there is an error during rendering, in a lifecycle method, or in the constructor of any child component. Method: componentDidCatch()
  5. super(props):
    When implementing the constructor for a React.Component subclass, you should call super(props) before any other statement. Otherwise, this.props will be undefined in the constructor, which can lead to bugs.
  • Commonly used lifecycle methods

  1. render()
    The only required method in a class component.
    When called, it should examine this.props and this.state and return.

  2. constructor()
    It is called before it is mounted.
    When implementing the constructor for a React.Component subclass, you should call super(props) before any other statement. Otherwise, this.props will be undefined in the constructor, which can lead to bugs.

  3. componentDidMount()
    It is invoked immediately after a component is mounted (inserted into the tree).
    Initialization that requires DOM nodes should go here.
    setState():
    You may call setState() immediately in componentDidMount(). It will trigger an extra rendering, but it will happen before the browser updates the screen. This guarantees that even though the render() will be called twice in this case, the user won’t see the immediate state.

  4. componentDidUpdate()
    It is invoked immediately after updating occurs.
    Network requests can be done here.

  5. componentWillUnmount()
    It is invoked immediately before a component is unmounted and destroyed.
    It will clean up any subscriptions that were created in componentDidMount(), including invalidating timers, canceling network requests.

  • Rarely used lifecycle methods:
  1. shouldComponentUpdate()
  2. static getDerivedStateFromProps()
  3. getSnapshotBeforeUpdate()
  4. componentDidCatch()
  • Legacy lifecycle methods:
  1. UNSAFE_componentWillMount()
  2. UNSAFE_componentWillReceiveProps()
  3. UNSAFE_componentWillUpdate()
  • Other APIs:
  1. setState()
    It enqueues changes to the component state and tells React that this component and its children need to be re-rendered with the updated state. It works as a request rather an immediate command to update the component. It will always lead to a re-render unless shouldComponentUpdate() returns false.
  2. foreceUpdate()
  • Class properties:
  1. defaultProps
    Is defined as a property on the component class itself, to set the default props for the class.
  2. displayName:
    Used in debugging messages. You don’t need to set it expicitly.
  • Instance properties:
  1. props
    this.props contains the props that were defined by the caller of this component.
  2. state
    The state contains data specific to this component that may change over time.
    Never mutate this.state directly, as calling setState() afterwards may replace the mutation you made. Treat this.state as if it were immutable.
    每一个React组件都有自己的state，其与props的区别在于state只存在组件的内部，props在所有实例中共享。
