Cache, cookies, and history:

  1. Cache
    Each time you access a file through your web browser, the browser caches it so it does not have to keep retrieving the same files or images from the remote web site each time you click Back or Forward.
    The downside is that the information on a website may have changed while your browser loads an older version.
  2. Cookie
    A cookie is a file created by a web browser, at the request of a website, that is stored on a computer. These files typically store user-specific information such as selections in a form, shopping cart contents, or authentication data.
    Cookies are small files that are downloaded to your computer when visiting a website. They can store previous activity on that site and user preferences so this information can be retrieved the next time you visit the same site. Cookies may also be used by 3rd parties to track a user’s browser history over a long period of time which can be considered a privacy concern.
  3. History
    A browser’s history is a log of sites that you visit.
  4. Differences
    Cookies is used to store information to track different characteristics related to user, while cache is used to make the loading of web pages faster.
    Cookies store information such as user perferences, while cache will keep resource files such as audio, video or flash files.
    Typically, cookies expire after some time, but cache is kept in the client’s machine until they are removed manually by the user.
  5. Extra note
    Session is used to store per-user information for the current web session on the server. It supports using a database server as the back-end store.
    Cookie should be used to store per-user information for the current web session or persistent information on the client, therefore client has control over the contents of a cookie.
    Cache object is shared between users in a single application. Its primary purpose is to cache data from a data store and should not be used as a primary storage. It supports automatic invalidation features.
    Application object is shared between users to store application-wide state and should be used accordingly.

    1. Let声明的变量的作用域只是外层块，而不是整个外层函数
    2. 形如for (let x…)的循环在每次迭代时都为x创建新的绑定

List vs. array：

  1. array是一段连续的存储结构。List则是不连续的存储结构，list的每个节点都有着一个next属性，这个属性则记录着他的下一个节点的地址。因为List是一个链表，所以我们需要从第一个元素开始逐个next到所需索引的元素。这是一个耗时的过程。
  2. 当不确定大小时，最好使用List代替array。当需要大量的查找操作时，最好使用array。当需要进行频繁的插入，删除操作时，最好使用list代替array。

OOP：

  1. 类只是具备了某些功能和属性的抽象模型，而实际应用中需要一个一个实体，也就是需要对类进行实例化，类在实例化之后就是对象。
  2. 封装性：将一个class的使用和实现分开，只保留部分接口和方法与外部联系
  3. 继承性：子类继承父类中的属性和方法，并可以添加新的属性和方法或者对部分属性和方法进行重写，增加了代码的可重用性。
  4. 多态性：多个子类虽然都继承父类的同一个方法，但是这些子类实例化的对象调用这些相同的方法后却可以获得完全不同的结果，这种技术就是多态性，增强了软件的灵活性。

Vue vs. event listener：

  1. V-on监听dom事件，并在触发时运行一些js代码
  2. 所有的vue.js事件处理方法和表达式都严格绑定在当前视图的ViewModel上，它不会导致任何维护上的困难。扫一眼html模板就能轻松定位js代码里的对应方法。ViewModel代码可以是非常纯粹的逻辑，和DOM完全解耦，更易于测试。当一个viewmodel被销毁，所有的事件处理器都会被自动删除。

Java abstract class feature:

  1. 抽象类主要用于被继承。多个类中若有共同的属性，则把共同的属性抽取出来，形成一个抽象类，然后继承该抽象类即可。
  2. 当一个抽象类中的所有成员只有抽象方法时，便是一个接口，可以用关键字interface表示。接口是用于被子类继承，再由子类根据自身需要自定义对应接口的。
  3. 使用接口可以实现多继承
  4. 类可以实现很多个接口，但是只能继承一个抽象类
  5. 接口中没有构造方法，也不能实例化接口对象
  6. 接口是一种极度抽象的类型，它比抽象类更加“抽象”。接口中不能使用implements,但是可以使用extends继承多个接口，是java中唯一可以多继承的地方。
